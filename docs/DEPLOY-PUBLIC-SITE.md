## 0. What you are deploying (and what you are NOT)

A **static, read-only** site that publishes verifyhash's independent-verification artifacts:

- the single-file, zero-dependency, **offline** verifier (`verify-vh-standalone.js`) + its published `sha256`,
- the companion offline **sealer** (`seal-vh-standalone.js`) + `sha256`,
- the **build-provenance manifest** (`BUILD-PROVENANCE.json`) that maps each bundle's hash back to its source modules,
- a hand-written **landing page** carrying the download links, the verify one-liner, the "reproduce the verifier from source" steps, and the honest trust boundary,
- the **prospect-facing docs** (CONFORMANCE, PILOT, KEY-LIFECYCLE, INDEPENDENT-VERIFICATION, …).

**NOT** in scope (deliberately): no backend, no uploads, no API, no database, no signing key, no
payment, no token. This is publication of already-public, git-tracked, byte-stable files. Nothing
dynamic runs. (The product's whole trust model is *offline* verification — a server that "verifies
for you" would weaken it. The one real server in the repo, `trustledger/server.js`, is a separate,
localhost-only broker demo that a human deploys behind their own access control; it is **not** part
of this site.)

## 4. nginx server block

Write `/etc/nginx/sites-available/verifyhash` (substitute `<YOUR_DOMAIN>`):

```nginx
server {
    listen 80;
    listen [::]:80;
    server_name <YOUR_DOMAIN> www.<YOUR_DOMAIN>;

    root /srv/verifyhash-public;     # <-- the dedicated webroot ONLY. NEVER the repo or /home.
    index index.html;

    autoindex off;                   # no directory listings
    server_tokens off;

    add_header X-Content-Type-Options "nosniff" always;
    add_header X-Frame-Options        "DENY"    always;
    add_header Referrer-Policy        "no-referrer" always;

    # Defense in depth: never serve dotfiles (.git, .env, …) even if one ever lands in the webroot.
    location ~ /\.  { deny all; return 404; }

    # The verifier + sealer: force download under the EXACT pinned filename so `sha256sum -c` works verbatim.
    location = /verify-vh-standalone.js {
        default_type application/javascript;
        add_header Content-Disposition 'attachment; filename="verify-vh-standalone.js"';
        add_header X-Content-Type-Options "nosniff" always;
    }
    location = /seal-vh-standalone.js {
        default_type application/javascript;
        add_header Content-Disposition 'attachment; filename="seal-vh-standalone.js"';
        add_header X-Content-Type-Options "nosniff" always;
    }

    location ~ \.sha256$            { default_type text/plain; }
    location = /build-provenance.json { default_type application/json; }
    location ~ \.md$               { default_type "text/markdown; charset=utf-8"; }

    location / { try_files $uri $uri/ =404; }
}
```

Enable + reload:

```bash
sudo ln -sf /etc/nginx/sites-available/verifyhash /etc/nginx/sites-enabled/verifyhash
sudo nginx -t && sudo systemctl reload nginx
```

## 6. Pre-go-live verification checklist

Run **all** of these; every one must pass before you announce the URL.

```bash
D=https://<YOUR_DOMAIN>

# a) the verifier downloads and its published hash checks out end-to-end
curl -fsS -o /tmp/v.js        "$D/verify-vh-standalone.js"
curl -fsS -o /tmp/v.js.sha256 "$D/verify-vh-standalone.js.sha256"
( cd /tmp && sha256sum -c v.js.sha256 )            # must say: verify-vh-standalone.js: OK

# b) the landing page renders and carries the hash + commands
curl -fsS "$D/" | grep -q 9358a657 && echo "landing OK"

# c) NONE of the forbidden paths are reachable (every one must be 404/403)
for p in .git/HEAD .git/config .env ../STRATEGY.md ../BACKLOG.md \
         ../build-loop.workflow.js ../../.claude/.credentials.json \
         ../docs/USAGE-BUDGET.json ../test/cli.claim.test.js; do
  code=$(curl -s -o /dev/null -w '%{http_code}' "$D/$p")
  echo "$code  $p"      # expect 403 or 404 for ALL of them — a 200 anywhere is a STOP-SHIP
done

# d) no directory listing
curl -s "$D/docs/" | grep -qi "index of" && echo "FAIL: autoindex on" || echo "no listing OK"
```

If any forbidden path returns `200`, **stop**, fix the nginx `root`/`location`, and re-run before going live.

## 7. Optional: a second, independent download origin

The verifier's trust anchor is its **hash**, so same-origin hosting is safe. But serving the verifier
from a *different* origin than the marketing/docs site makes the "don't trust the vendor" pitch literally
true at the network layer: a reviewer who distrusts `<YOUR_DOMAIN>` can fetch the bundle + its `sha256`
from `verify.<YOUR_DOMAIN>` (a separate server block, or ideally a different host/CDN) and cross-check
the hash against `BUILD-PROVENANCE.json`. **Nice-to-have, not a launch blocker.** If you do it, repeat
§3a/§3b/§4 for that origin serving only the four files: `verify-vh-standalone.js` (+`.sha256`),
`build-provenance.json`.

## 8. Guardrail boundary — division of labor

- **The verifyhash loop (unprivileged) BUILDS artifacts in-tree** (the `verifier/dist/*` bundles, this
  runbook, optionally a staged `public/`) as **local commits**. It never pushes, deploys, binds a port,
  holds a key, or touches anything outside its own home.
- **You (this privileged agent / the human) perform the outward-facing steps**: copy to `/srv`, configure
  nginx, request TLS, set DNS. That is the human action the loop is forbidden from taking.
- **Nothing here is a deployed service.** Static publication of already-public, git-tracked files crosses
  no no-deploy / no-funds / no-token line. There is no key, no payment, no data custody, no token issuance.
- If you later want the **TrustLedger broker demo** (`trustledger/server.js`) reachable, that is a
  *separate* decision: it binds `127.0.0.1` by default and ingests trust-account data — put it behind your
  own auth/nginx, never co-host it with this static site, and treat it as its own needs-human deployment.
