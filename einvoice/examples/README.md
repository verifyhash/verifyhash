# einvoice onboarding examples — a 5-minute quickstart

This directory is a hands-on walkthrough of the `einvoice` conformance checker:
take a **broken** invoice, run the tool, **read the report**, apply the fixes
it names, and end with a **corrected** invoice that passes. Every file here is
real — the reports are generated by the actual engine (see
[`gen_examples.py`](../gen_examples.py)), not hand-written, and a test
(`test_examples.py`) fails the build if any committed report drifts from what
the tool emits.

No dependencies, no network. If you have Python 3 and this repository, you can
reproduce every command below.

## What the tool does (and does not) do

`einvoice` validates electronic invoices against the **EN 16931** semantic
model and the German **XRechnung** CIUS (the `BR-DE-*` rules) — the format
public-sector suppliers in Germany must send. It parses UBL and CII (Factur-X /
ZUGFeRD PDFs too) with the Python standard library only, and reports each
violated business rule with its EN 16931 term ids (`BT-` fields, `BG-` groups)
and a concrete fix hint.

Honest limits: it checks the **business rules** (required fields, code lists,
VAT-category and monetary-total arithmetic, the XRechnung national rules), not
XSD schema validity, and it is not the official KoSIT validator — treat it as a
fast pre-flight that catches the mistakes that trip up most first submissions,
then run your buyer's official validator before you file.

Findings carry a severity. Only **fatal** findings make an invoice invalid
(exit code 1), mirroring the official Schematron `flag` semantics. `warning`
and `information` findings are advisory and do **not** fail the build.

## Example: `01-missing-fields/`

A supplier exported a UBL invoice but two required things are missing. The file
[`01-missing-fields/broken.xml`](01-missing-fields/broken.xml) is a minimal
mutation of a real, valid XRechnung test document
(`corpus/vendored/valid/xr-01.01a_ubl.xml`) with exactly two elements removed,
so the *only* reason it fails is those two removals.

### 1. Here is the broken invoice

`broken.xml` is missing:

- the **Buyer reference** (`BT-10`, `<cbc:BuyerReference>`) — the
  *Leitweg-ID* / routing id a German public buyer requires, and
- the **SELLER CONTACT** group (`BG-6`, `<cac:Contact>` under the supplier
  party).

### 2. Run the checker

```console
$ python3 -m einvoice.report examples/01-missing-fields/broken.xml --format json
```

(run from the `einvoice/` directory). It exits **1** and prints a JSON report;
the exact bytes are committed next to the invoice as
[`01-missing-fields/report.json`](01-missing-fields/report.json). Add
`--pretty` for indented output, or `--format text` for a human summary.

### 3. Read the findings and fix them

The report lists three findings — two `fatal` (which is why the invoice is
invalid) and one advisory `information`:

| rule | severity | terms | what it means / how to fix |
|------|----------|-------|-----------------------------|
| `BR-DE-2`  | **fatal** | `BG-6`  | *The group 'SELLER CONTACT' (BG-6) must be transmitted.* Add a `<cac:Contact>` under `cac:AccountingSupplierParty/cac:Party` (name, phone and/or e-mail). |
| `BR-DE-15` | **fatal** | `BT-10` | *The element 'Buyer reference' (BT-10) must be transmitted.* Add a non-empty `<cbc:BuyerReference>` (the buyer's routing id / Leitweg-ID). |
| `BR-DE-TMP-32` | information | `BT-72`, `BG-14`, `BG-26` | Advisory: state a delivery/service date. Does **not** fail the build; left as-is here so the example stays a minimal two-field fix. |

Each record carries the same information in machine-readable form: `rule`,
`severity`, `message`, the `field`/`location` XPath, the EN 16931 `terms`, and a
`fix_hint` string you can surface directly to a user. For a full remediation
write-up of any rule, run `python3 -m einvoice.report --explain BR-DE-15`.

### 4. Here is the corrected invoice

[`01-missing-fields/fixed.xml`](01-missing-fields/fixed.xml) is `broken.xml`
with those two elements restored (it is byte-for-byte the original valid corpus
document). Re-run the checker on it:

```console
$ python3 -m einvoice.report examples/01-missing-fields/fixed.xml --format json
```

It now exits **0** and reports `"valid":true` with `"fatal_count":0`. The two
`BR-DE-*` fatals are gone — the invoice would pass this pre-flight.

## Regenerating and checking the reports

The committed `report.json` files are produced by the real engine, so they
cannot silently disagree with the tool:

```console
$ python3 gen_examples.py          # rewrite every report.json from the engine
$ python3 gen_examples.py --check  # exit 1 if any committed report is stale
$ python3 test_examples.py         # assert broken fails, fixed passes, no drift
```

`test_examples.py` re-runs the engine on every `broken.xml`, asserts the
committed `report.json` matches live output field-for-field, asserts each
`broken.xml` fires at least one fatal finding, and asserts each `fixed.xml`
reports `valid:true` with zero fatal findings.
