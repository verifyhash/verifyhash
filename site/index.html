verifyhash

Cryptographic provenance you can check yourself — offline, with zero trust in us. A sealed artifact carries its own proof. You re-derive it from the bytes on your own machine; you don't take our word, and you don't call our server (there isn't one).

1. Get the verifier (one file, zero dependencies)

curl -O https://verifyhash.com/verify-vh-standalone.js
curl -O https://verifyhash.com/verify-vh-standalone.js.sha256
sha256sum -c verify-vh-standalone.js.sha256     # macOS: shasum -a 256 -c …

Published SHA-256:

793125a8de58cea35e1e1cd1bb7d1b315c93c59f9045af8a4b1152a443b452ac

It opens no network, needs only Node core, and exits 0 = accepted / 3 = rejected.

2. Verify a sealed packet someone sent you

node verify-vh-standalone.js <packet.vhevidence.json> --vendor 0x<the address the producer told you to pin>

It recomputes every hash from the packet's own bytes and confirms the signer address you pinned.

3. Don't trust our checksum — reproduce the verifier from source

Read the source in verifier/, then rebuild the bundle in memory and confirm it is byte-for-byte what we published — offline, writing nothing:

node verifier/build-standalone.js --check        # exit 0 = every bundle, sidecar & source hash MATCH

The build-provenance manifest maps each published hash back to the exact, individually-hashed source modules it inlines, so you can root trust in source you read.

4. The honest trust boundary — read before you rely on it

A seal proves tamper-evidence + signer-pin: the bytes are unaltered and were signed by the key whose address you pinned. It is NOT a trusted "sealed at time T" (that needs an external timestamp authority or anchor), and reproducing the verifier proves build integrity — that our bundle is faithfully built from its source — NOT that the source's logic is correct (read it, and run the conformance corpus, for that), and NOT a trusted timestamp or identity. Verification is free; producing seals is the paid surface.
